# Especificación de Requisitos: Integración de Servidores MCP (Model Context Protocol)
## Context.ai — Módulo de Conocimiento Dinámico

> **Versión**: 1.0  
> **Estado**: Diseño — En análisis de requisitos  
> **Módulo**: Knowledge / Interaction  

---

## 1. Propósito y Contexto de Negocio

### 1.1 Visión General
La integración de **Servidores MCP (Model Context Protocol)** permite a Context.ai expandir su base de conocimientos más allá de los documentos estáticos (PDFs, Markdown). Mediante este protocolo estándar y de código abierto (creado por Anthropic y adoptado por la industria), el asistente RAG podrá conectarse en tiempo real a herramientas empresariales externas, bases de datos y APIs para extraer contexto en el momento exacto en que el usuario hace una consulta.

### 1.2 Problema que resuelve
| Problema | Impacto | Solución Propuesta (MCP) |
|----------|---------|--------------------------|
| **Información Desactualizada** | El usuario consulta una política de vacaciones en un PDF de hace 3 meses, ignorando cambios recientes en el sistema de RRHH. | Conectar un servidor MCP del software de RRHH para leer el saldo de días en tiempo real. |
| **Silos de Información** | El conocimiento técnico vive en Jira o GitHub, obligando a duplicar esa información en documentos para Context.ai. | Context.ai puede buscar directamente en tickets de Jira o PRs de GitHub vía herramientas MCP. |
| **Fricción Operativa** | Mantenimiento manual pesado: los administradores deben estar subiendo y borrando documentos constantemente. | Acceso en vivo ("Live RAG"): menos PDFs subidos, más conexiones directas a las fuentes de la verdad. |

### 1.3 Propuesta de valor
> **"Pregúntale a toda tu empresa, no solo a tus PDFs."**
Convertir a Context.ai en un agente inteligente capaz de decidir cuándo buscar en la base vectorial (Pinecone) y cuándo ejecutar herramientas externas vía MCP para proporcionar respuestas precisas, contextualizadas y en tiempo real, respetando los permisos de acceso de cada sector.

## 2. Requisitos Funcionales (RF) y Criterios de Aceptación

### RF-1: Gestión de Servidores MCP (Admin UI)
**Descripción**: Los administradores deben poder conectar y configurar nuevos servidores MCP en la plataforma.
* **Criterios de Aceptación**:
  * [ ] Nueva vista en el panel Admin: `Configuración > Servidores MCP`.
  * [ ] Formulario para registrar un servidor MCP especificando: Nombre, Tipo de transporte (SSE/HTTP), URL del endpoint y Credenciales (API Keys/Tokens).
  * [ ] Botón para "Probar conexión" (Ping) que verifique si el servidor MCP responde y liste las herramientas (`tools`) que expone.
  * [ ] Capacidad para Activar/Desactivar servidores MCP globalmente.

### RF-2: Asignación de MCP por Sector
**Descripción**: No todos los sectores deben tener acceso a todos los servidores MCP (Ej: El sector "Ventas" no debe consultar la base de datos de "Recursos Humanos").
* **Criterios de Aceptación**:
  * [ ] En la configuración de un `Sector`, debe existir una sección para vincular servidores MCP específicos.
  * [ ] Un servidor MCP puede estar vinculado a múltiples sectores (N:M).
  * [ ] Cuando un usuario hace una consulta en un sector, el LLM solo tendrá acceso a las herramientas de los servidores MCP vinculados a ese sector.

### RF-3: Ejecución de Herramientas Dinámicas (Genkit + MCP Client)
**Descripción**: El flujo de conversación debe cargar dinámicamente las herramientas MCP y proporcionárselas a Gemini para que decida cuándo usarlas.
* **Criterios de Aceptación**:
  * [ ] El backend debe inicializar clientes MCP (`@modelcontextprotocol/sdk`) en tiempo de ejecución.
  * [ ] Las herramientas expuestas por el servidor MCP deben mapearse dinámicamente como `ai.defineTool` en Genkit.
  * [ ] El LLM (Gemini 2.5 Flash/Pro) determinará de forma autónoma si responde con los documentos RAG tradicionales o si invoca una herramienta MCP según la pregunta del usuario.

### RF-4: Transparencia y Citas en la UI
**Descripción**: El usuario debe saber si la respuesta proviene de un PDF o de una herramienta externa en vivo.
* **Criterios de Aceptación**:
  * [ ] En el Frontend, mientras la IA "piensa", se debe mostrar un indicador: *“Consultando [Nombre de Herramienta MCP]...”*
  * [ ] La respuesta final debe citar el origen. Ej: "Fuente: Ticket de Jira HR-123 (Vía MCP)".

## 3. Requisitos No Funcionales (RNF)

* **RNF-1: Seguridad de Credenciales**: Las contraseñas, API Keys o Tokens necesarios para conectarse a los servidores MCP externos deben guardarse cifrados (encriptación simétrica) en la base de datos de PostgreSQL, y desencriptarse solo en memoria al inicializar el cliente MCP.
* **RNF-2: Arquitectura de Transporte (SSE)**: Dado que el backend (NestJS) se despliega en contenedores (Cloud Run/Docker), se priorizará el transporte MCP vía **SSE (Server-Sent Events) sobre HTTP**, descartando el transporte por `stdio` (que requeriría instalar binarios locales en el contenedor de NestJS).
* **RNF-3: Tolerancia a Fallos**: Si un servidor MCP externo está caído o tarda más de 10 segundos en responder, la herramienta debe devolver un error controlado al LLM ("Servicio no disponible") para que el asistente pueda decirle al usuario: "No pude acceder a X en este momento", sin bloquear (crashear) todo el chat.
* **RNF-4: Rendimiento (Streaming)**: Las llamadas a herramientas externas añaden latencia (Multi-turn Tool Calling). Es obligatorio que la UI implemente *Streaming* fluido para que el usuario no perciba que el sistema se ha congelado mientras la IA interactúa con el MCP.

## 4. Arquitectura y Modelo de Datos

### 4.1 Nuevas Entidades en Base de Datos (TypeORM)

**Entidad: `McpServer`**
* `id` (UUID, PK)
* `name` (String) - Ej: "Jira Interno", "Notion Empresa"
* `transportType` (Enum) - `SSE` o `STDIO` (enfocados en SSE)
* `endpoint` (String) - URL del servidor MCP
* `encryptedConfig` (Text) - JSON encriptado con headers o tokens de autenticación
* `status` (Enum) - `ACTIVE`, `INACTIVE`, `ERROR`

**Entidad Intermedia: `SectorMcpServer`** (Tabla Join N:M)
* `sectorId` (FK -> Sectors)
* `mcpServerId` (FK -> McpServers)

### 4.2 Flujo de Arquitectura (Agentic RAG Flow)

```text
[Frontend] Usuario: "¿Cuáles son los tickets abiertos para el proyecto Alpha?"
   │
   ▼
[NestJS] 1. Identifica el `sectorId` actual.
         2. Consulta en BD los `McpServers` activos para ese sector.
         3. Inicializa los MCP Clients (vía SSE) y hace un `listTools()`.
   │
   ▼
[Genkit] 4. Convierte las herramientas MCP a Genkit Tools (`ai.defineTool`).
         5. Inicia generación con Gemini 2.5 pasándole: 
            - Prompt / Historial
            - Contexto Pinecone (RAG normal)
            - Herramientas MCP
   │
   ▼
[Gemini] 6. (Function Calling) Decide que necesita ejecutar la herramienta `jira_search`.
   │
   ▼
[MCP Client] 7. NestJS ejecuta la herramienta contra el Servidor MCP de Jira externo.
             8. Devuelve el JSON con los tickets a Gemini.
   │
   ▼
[Gemini] 9. Sintetiza la respuesta final en lenguaje natural basándose en los datos de Jira.
   │
   ▼
[Frontend] Recibe respuesta (Streaming) indicando la fuente.

### 5. Restricciones Técnicas
SDK Oficial: Se deberá usar estrictamente el paquete oficial de TypeScript: @modelcontextprotocol/sdk.

Compatibilidad Genkit: Actualmente, Genkit tiene soporte nativo para Model Context Protocol a través de plugins (ej. genkitx-mcp o adaptadores personalizados). El equipo debe crear una fábrica o adaptador que lea el esquema JSON devuelto por MCP listTools() y lo convierta a esquemas de Zod compatibles con Genkit.

Redes y VPC: Si el servidor MCP está alojado en la red interna de la empresa, el contenedor de NestJS (Context.ai) debe estar configurado en la misma VPC (Virtual Private Cloud) para alcanzar la IP interna del servidor MCP.

### 6. Plan de Implementación Propuesto
Fase 1: Infraestructura y CRUD (Semana 1)
Crear la migración para las tablas mcp_servers y sector_mcp_servers.

Implementar servicio de cifrado/descifrado (AES-256-GCM) para encryptedConfig.

Desarrollar endpoints del backend y vistas en el Admin UI para registrar, probar y asignar servidores MCP a sectores.

Fase 2: Motor MCP a Genkit (Semana 2 - Crítica)
Crear el servicio McpIntegrationService en NestJS que:

Se conecte al servidor remoto usando SSEClientTransport.

Extraiga las herramientas disponibles.

Genere dinámicamente los envoltorios (tools de Genkit) usando Zod inferido.

Integrar esto en el caso de uso query-assistant.use-case.ts, inyectando las herramientas antes de llamar al modelo.

Fase 3: Frontend y UX (Semana 3)
Actualizar el componente MessageList o StructuredResponse para mostrar el indicador temporal (Ej: "🤖 Usando herramienta: notion_search...").

Actualizar el SourceCard para que soporte no solo documentos RAG (PDFs), sino metadatos de MCP (Ej: Icono de enlace, URL, nombre del servidor).

Pruebas End-to-End conectando un servidor MCP mockeado simple (ej: un MCP de clima o memoria).

### 💡 Consejos de Arquitectura para tu Stack:
Dado que usas **Genkit** y **NestJS** en contenedores (probablemente Cloud Run o ECS):
1. **Evita usar MCP vía `stdio`** (comandos de terminal `npx...`). Es un dolor de cabeza enorme instalar binarios como Python, Node, o Go dentro del mismo contenedor de producción de NestJS solo para correr servidores MCP.
2. **Apuesta todo por MCP vía `SSE` (Server-Sent Events)**: Esto permite que despliegues los servidores MCP como microservicios separados (en otros contenedores o servidores) y tu Context.ai simplemente se conecte a ellos por HTTP/URL. En el PRD ya he dejado estipulada esta restricción técnica para ahorrarles problemas a futuro.
