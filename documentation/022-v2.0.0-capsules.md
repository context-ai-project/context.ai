# Especificación de Requisitos V2: Generación de Cápsulas Multimedia
## Context.ai — Módulo de Contenido Multimedia con IA

> **Versión**: 2.2 (Arquitectura Programática + Serverless Async)  
> **Fecha**: Marzo 2026  
> **Estado**: Diseño — En análisis de requisitos  
> **Prerequisito**: v1.3.0 (Admin + RBAC + Sectores + RAG Chat)  
> **Caso de Uso base**: UC3 — Generar Cápsulas Multimedia (ver `002-use-cases.md`)

---

## 1. Propósito y Contexto de Negocio

### 1.1 Visión General

La **Generación de Cápsulas Multimedia** es la evolución natural de Context.ai hacia un sistema de onboarding y formación más inmersivo. Mientras la v1 se centró en la ingesta documental y el chat RAG (texto), la v2 transforma el conocimiento escrito en **contenido audiovisual explicativo** — videos y audios generados por IA — que facilitan el aprendizaje y la retención del conocimiento.

### 1.2 Problema que resuelve

| Problema | Impacto | Solución V2 |
|----------|---------|-------------|
| Documentos extensos y poco atractivos | Baja tasa de lectura completa por nuevos integrantes | Videos tipo "presentación dinámica" generados por IA |
| No todos los empleados aprenden leyendo | Brecha de comprensión en perfiles no técnicos | Audio-guías con narración profesional (TTS) |
| RRHH no tiene recursos para grabar videos | Tiempo excesivo en producción de material de onboarding | Generación automatizada de guiones, imágenes, audio y ensamblaje |
| Contenido desactualizado rápidamente | Los manuales se actualizan pero los videos no | Crear nuevas cápsulas desde documentos actualizados a muy bajo coste |

### 1.3 Propuesta de valor

> **"De un documento de 50 páginas a una cápsula de aprendizaje audiovisual en minutos, con un clic."**

El módulo permite a los usuarios con rol **admin** o **manager** seleccionar documentos previamente subidos a un sector de conocimiento, redactar un texto introductorio personalizado, y — mediante inteligencia artificial — generar automáticamente:

1. Un **guion narrativo estructurado por escenas** a partir del contenido documental (IA generativa con Gemini 1.5 Flash).
2. Un **audio profesional** de la narración (Text-to-Speech con ElevenLabs).
3. **Imágenes ilustrativas** para cada escena (IA generativa con Vertex AI / Imagen 3).
4. Un **video final ensamblado programáticamente** (Shotstack API) combinando el audio, las imágenes, transiciones y textos en pantalla.

Los empleados con rol **user** tendrán acceso exclusivamente a la **visualización y reproducción** de las cápsulas generadas, dentro de los sectores a los que tienen acceso.

3. Requisitos Funcionales (RF)
RF-V2-1: Generación de Guion Estructurado con IA
Descripción: El sistema utilizará Gemini (Genkit) para generar un guion estructurado a partir de los documentos seleccionados.
Si el tipo es Video, el LLM debe retornar una estructura JSON de "escenas" (cada escena contiene el texto a narrar, el prompt visual para la imagen de fondo, y un texto corto para sobreimprimir).
Si el tipo es Audio, retorna un texto narrativo continuo.

Flujo técnico (Video):

Documentos RAG → Construir prompt pidiendo JSON de escenas
  → Gemini 1.5 Flash genera Array de { textToNarrate, visualPrompt, titleOverlay }
    → Retornar al frontend para revisión/edición

RF-V2-2: Generación de Audio (Text-to-Speech)
Descripción: A partir del guion narrativo, se sintetiza un audio profesional usando la API de ElevenLabs en formato MP3.

RF-V2-3: Generación de Visuales y Ensamblaje Programático de Video
Descripción: El sistema generará un video combinando dinámicamente imágenes, audio y texto, sin depender de costosos modelos de video generativo (Sora/Veo).

Criterios de Aceptación:

[ ] El backend itera sobre las escenas del guion y llama a Gemini Imagen 3 para generar las ilustraciones correspondientes a cada visualPrompt.

[ ] Las imágenes generadas se suben temporalmente a Google Cloud Storage.

[ ] Se genera la pista de audio completa usando ElevenLabs.

[ ] El backend construye un objeto JSON (Timeline) definiendo la secuenciación de pistas: Audio de fondo, secuencia de imágenes (ej. 8 segundos cada una con efecto Ken Burns sutil), y superposición de textos (titleOverlay).

[ ] Se envía el Timeline a la API de Shotstack para su renderizado.

[ ] El sistema hace polling del estado del render en Shotstack.

[ ] Al finalizar, se descarga el MP4 final, se almacena en el bucket GCS de la plataforma, y se notifica la disponibilidad.

Flujo técnico:

Guion Estructurado (JSON)
  ├── 1. Imágenes: IImageGenerator (Vertex AI Imagen 3) → subida a GCS temporal
  ├── 2. Audio: IAudioGenerator (ElevenLabs) → subida a GCS temporal
  └── 3. Ensamblaje: IVideoRenderer (Shotstack API) enviando Timeline JSON
        → Polling estado
          → Descarga de MP4 final a GCS definitivo

RF-V2-4: Gestión de Cápsulas (CRUD y Estados)
Estados del pipeline asíncrono (CapsuleStatus):
DRAFT → QUEUED (Cloud Tasks) → GENERATING_ASSETS (imágenes/audio) → RENDERING (Shotstack) → COMPLETED → ACTIVE (o FAILED).

Manejo de fallos: Al separar la generación de imágenes de la de video, un fallo en el renderizado de Shotstack no requiere volver a pagar/generar las imágenes o el audio. El sistema (vía reintentos de Cloud Tasks o intervención manual) permite reintentar solo la etapa fallida.

4. Requisitos No Funcionales (RNF)
RNF-V2-1: Rendimiento y Costes
Proceso,Tiempo estimado,Coste estimado
Generación de Guion (Gemini 1.5),< 10 segundos,Fracciones de céntimo
Imágenes (Imagen 3),< 15 segundos (paralelizable),~$0.03 por imagen
Audio (ElevenLabs),< 15 segundos,Según plan (económico)
Renderizado (Shotstack),1-3 minutos,~$0.02 - $0.05 / minuto de video

Esta arquitectura reduce el coste de generación de video de dólares a apenas unos céntimos por minuto, permitiendo mayor volumen sin preocuparse estrictamente por cuotas pequeñas.

RNF-V2-2: Procesamiento Asíncrono 100% Serverless
La generación de contenido multimedia requiere un procesamiento asíncrono desacoplado para no bloquear el hilo de NestJS ni hacer timeout en la petición HTTP original del cliente.
Para mantener un stack económico y nativo de la nube (sin necesidad de pagar Redis o CPU inactiva):

Se utilizará Google Cloud Tasks como orquestador asíncrono en lugar de herramientas de polling de colas (como BullMQ).

Al iniciar la generación, el backend encolará una tarea en Cloud Tasks.

Cloud Tasks realizará una petición HTTP (POST a un endpoint interno protegido, ej: /api/v1/capsules/internal/process/:id) para despertar a Cloud Run de forma asíncrona y ejecutar el pipeline.

El frontend realizará polling al endpoint /api/v1/capsules/:id/status para actualizar la UI.

RNF-V2-3: Almacenamiento
Audios (MP3) y Videos (MP4) se guardan en un bucket privado de GCS y se entregan a los usuarios mediante URLs firmadas (Signed URLs) de 1 hora de caducidad.

RNF-V2-4: Arquitectura de Abstracción (Clean Architecture)
Las dependencias externas se encapsulan detrás de interfaces de dominio en context-ai-api/src/modules/capsules/domain/services/:

// 1. Generador de imágenes para las escenas
export interface IImageGenerator {
  generateImage(prompt: string): Promise<Buffer>;
}

// 2. Generador de voz
export interface IAudioGenerator {
  generateAudio(text: string, voiceId: string): Promise<Buffer>;
}

// 3. Renderizador de video (Ensamblador)
export interface IVideoRenderer {
  renderVideo(timeline: VideoTimelineRequest): Promise<string>; // Retorna renderId
  getRenderStatus(renderId: string): Promise<RenderStatusResponse>;
}

5. Restricciones Técnicas

Componente,Tecnología,Justificación
Imágenes (Visuales),Google Vertex AI (Imagen 3),"Generación de ilustraciones coherentes, integrado en el GCP actual."
Audio (TTS),ElevenLabs API,Voces ultrarrealistas.
Ensamblaje Video,Shotstack API,Edición de video cloud-native vía JSON. Evita la complejidad y alto consumo de CPU de usar FFmpeg en el propio backend.
Backend Async,Google Cloud Tasks,Procesamiento asíncrono serverless (webhook-based). Evita el coste fijo de Redis y permite que Cloud Run escale a 0 (ahorro de CPU).

6. Plan de Implementación Propuesto
La implementación se divide en dos bloques, de menor a mayor complejidad.

🔊 BLOQUE A — Cápsulas de Audio (Fases 1-4)
Se establece el CRUD, la base de datos, los permisos RBAC, el almacenamiento en GCS (Signed URLs), el generador de guiones en Genkit, el adaptador de ElevenLabs y la orquestación básica con Cloud Tasks para que no haya timeouts.

Entregable Bloque A:
Cápsulas de audio 100% funcionales (seleccionar docs → generar guion → generar audio → publicar). Tiempo estimado: 11-15 días.

🎬 BLOQUE B — Cápsulas de Video (Fases 5-7)
Fase 5: Backend — Pipeline de Video con Shotstack y Cloud Tasks (4-5 días)
Definición de interfaces IImageGenerator y IVideoRenderer.

Implementación de VertexImageGeneratorService usando el SDK de @google-cloud/vertexai (modelo Imagen 3).

Implementación de ShotstackRendererService usando shotstack-sdk.

Creación de la lógica que transforma el array de escenas (GuionScene[]) en un objeto JSON Timeline de Shotstack (Track de imágenes, Track de audio de fondo, Track de textos).

Configuración del flujo Google Cloud Tasks:

Al confirmar generación, el endpoint principal encola una tarea en Cloud Tasks (Payload: ID de la cápsula) y retorna status QUEUED al frontend.

Creación del endpoint interno protegido (Webhook) POST /internal/capsules/:id/process.

Al recibir el webhook, se orquesta el pipeline:

Descarga/Generación de assets (Audio + N imágenes en paralelo).

Subida temporal a GCS (Shotstack requiere URLs públicas o firmadas).

Envío de petición POST a Shotstack para iniciar el renderizado.

Nota: Si el renderizado de Shotstack es muy largo, se puede crear un webhook de retorno (Shotstack Webhook) que avise al backend cuando termine, en lugar de bloquear el webhook de Cloud Tasks haciendo polling.

Descarga del MP4 final al GCS privado y limpieza de temporales.

Fase 6: Frontend — Extensión del Wizard para Video (2-3 días)
Modificar Step 1 para añadir opción "🎬 Video" (eliminando la opción "Ambos" para simplificar la UX).

Actualizar la interfaz del Step 2 cuando es Video:

El guion ya no es un textarea simple, sino una lista editable de "Escenas" (cada una con texto de narración y descripción de imagen).

Actualizar el reproductor final en CapsulePlayerView para soportar video.

Polling del progreso (QUEUED -> GENERATING_ASSETS -> RENDERING -> COMPLETED).

Fase 7: Integración y Testing (2-3 días)
    Tests de integración para la orquestación asíncrona simulando el comportamiento de Cloud Tasks.
    Gestión de errores y reintentos (configurar la política de backoff exponencial en la cola de Cloud Tasks).
    Despliegue.

Entregable Bloque B:
Generación de video tipo "diapositivas dinámicas" funcionando. Escalable, serverless, sin bloqueos de peticiones HTTP, sin costo base de bases de datos de colas y extremadamente económico comparado con modelos puros de video generativo. Tiempo estimado: 8-11 días.

7. Cambios en el Modelo de Datos
Las tablas capsules sufrirán un pequeño ajuste para soportar la estructura de escenas para el video y el seguimiento asíncrono:

    type: ENUM('video', 'audio') (se remueve 'both')

    script: Almacenará texto plano para audios, o un JSON array de escenas para videos.

    generation_metadata: Almacenará el cloudTaskId (ID de la tarea asíncrona) y el renderId de Shotstack para permitir la trazabilidad y la reanudación del proceso en caso de fallos.

8. Glosario Actualizado

Término,Definición
Cápsula,Unidad de contenido multimedia (video o audio) generada por IA.
Cloud Tasks,Servicio de Google Cloud que gestiona colas de tareas distribuidas y entrega webhooks de forma asíncrona. Reemplaza a BullMQ/Redis.
ElevenLabs,Plataforma de síntesis de voz (TTS).
Imagen 3,Modelo fundacional de Google para generación de imágenes a partir de texto. Se usa para crear los fondos visuales del video.
Shotstack,"API de edición de video cloud. Permite componer videos uniendo imágenes, audio y texto mediante un archivo JSON (Timeline)."
Timeline JSON,"Estructura de datos que describe las capas, tiempos y transiciones que Shotstack debe renderizar para crear el video final."
