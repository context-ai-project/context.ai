## 1. Propósito y Contexto de Negocio

### 1.1 Visión General
El **Módulo de Análisis de Sentimiento y Feedback** es una evolución analítica para Context.ai. Su objetivo es rastrear qué tan útil es la documentación subida al sistema midiendo la satisfacción del usuario frente a las respuestas del asistente (RAG). Combina retroalimentación explícita (calificaciones directas del usuario) con retroalimentación implícita (análisis de IA sobre el tono del mensaje) para generar insights accionables para el equipo de Recursos Humanos o Administradores.

### 1.2 Problema que resuelve
| Problema | Impacto | Solución Propuesta |
|----------|---------|--------------------|
| **Puntos ciegos documentales** | RRHH no sabe si los manuales subidos resuelven realmente las dudas de los empleados. | Trazabilidad exacta entre las respuestas insatisfactorias y el documento fuente que las generó. |
| **Falta de métricas de calidad** | No hay forma de medir la eficacia del sistema RAG más allá de la cantidad de mensajes. | Sistema de calificación (👍/👎) por cada respuesta del asistente. |
| **Frustración silenciosa** | Los usuarios rara vez dejan quejas formales cuando un sistema interno no les ayuda; simplemente se frustran. | La IA analiza asíncronamente el tono de las preguntas para detectar confusión o frustración automáticamente. |

### 1.3 Propuesta de valor
> **"Transformar el chat de los empleados en un auditor continuo de la calidad documental de la empresa."**
Permitir a los administradores visualizar un "Dashboard de Salud del Conocimiento" que identifique de forma proactiva qué documentos están desactualizados, son confusos o están incompletos, basándose en las reacciones reales de los empleados.

## 2. Requisitos Funcionales (RF) y Criterios de Aceptación

### RF-1: Feedback Explícito del Usuario (UI)
**Descripción**: El sistema permitirá al usuario calificar la utilidad de cada respuesta proporcionada por el asistente RAG.
* **Criterios de Aceptación**:
  * [ ] Cada burbuja de respuesta del asistente en el chat debe incluir botones discretos de 👍 (Útil) y 👎 (No útil).
  * [ ] Al hacer clic en 👎, el sistema debe desplegar un formulario opcional (Popover/Dialog) para categorizar el error (Ej: "Información desactualizada", "Incompleto", "Confuso") y permitir texto libre.
  * [ ] El usuario puede cambiar su voto en cualquier momento.
  * [ ] El feedback se guarda en la base de datos asociado al `messageId` específico.

### RF-2: Análisis de Sentimiento Implícito (IA Asíncrona)
**Descripción**: El sistema evaluará el tono y la intención del mensaje del usuario utilizando un modelo de IA (Gemini Flash vía Genkit) para detectar frustración o confusión, sin que el usuario tenga que calificar explícitamente.
* **Criterios de Aceptación**:
  * [ ] Cada mensaje enviado por el usuario será procesado en segundo plano para asignarle una etiqueta de sentimiento (`POSITIVE`, `NEUTRAL`, `NEGATIVE`, `CONFUSED`, `FRUSTRATED`).
  * [ ] El modelo de IA asignará un score de certeza (-1.0 a 1.0).
  * [ ] El proceso NO debe añadir latencia a la respuesta del chat RAG (debe ser 100% asíncrono).

### RF-3: Trazabilidad Documental (RAG Tracking)
**Descripción**: El sistema debe registrar qué documentos exactos se usaron para generar cada respuesta de la IA.
* **Criterios de Aceptación**:
  * [ ] Al recuperar fragmentos de Pinecone, el backend debe extraer los UUIDs de los `KnowledgeSource` correspondientes.
  * [ ] La entidad del mensaje guardará un array con los IDs de los documentos (`usedSourceIds`) que alimentaron esa respuesta.

### RF-4: Dashboard de "Salud del Conocimiento"
**Descripción**: Una nueva vista en el panel de administración donde RRHH evaluará las métricas recolectadas.
* **Criterios de Aceptación**:
  * [ ] Mostrar un gráfico de dona/radar con la distribución general de sentimientos.
  * [ ] Mostrar una tabla ("Top Documentos a Revisar") ordenada por la cantidad de `DISLIKES` o sentimientos negativos asociados.
  * [ ] Mostrar una lista de comentarios/quejas directas de los usuarios filtrables por sector.
  * [ ] Restricción de acceso: Solo visible para usuarios con roles `admin` y `manager`.

## 3. Requisitos No Funcionales (RNF)

* **RNF-1: Rendimiento UI/UX**: El guardado del feedback explícito (👍/👎) debe reflejarse instantáneamente en la interfaz de usuario (Optimistic UI update) mientras se procesa la petición de red.
* **RNF-2: Procesamiento No Bloqueante**: El análisis de sentimiento de Genkit debe ejecutarse a través de Eventos de NestJS (`EventEmitter2`) o Google Cloud Tasks. Bajo ninguna circunstancia debe retrasar la respuesta al usuario en el endpoint de chat.
* **RNF-3: Privacidad**: En el Dashboard de RRHH, los reportes de sentimiento y feedback no mostrarán el nombre del usuario que hizo la pregunta, enfocándose únicamente en el "Documento" y el "Sector", garantizando el anonimato y la seguridad psicológica del empleado.
* **RNF-4: Eficiencia de Costos**: El análisis de sentimiento asíncrono se realizará utilizando `gemini-2.5-flash`, priorizando la velocidad y el bajo coste computacional sobre modelos Pro.

## 4. Arquitectura y Modelo de Datos

### 4.1 Cambios en Base de Datos (PostgreSQL)
Se extenderá la entidad `Message` existente (en `src/modules/interaction/domain/entities/message.entity.ts`) con los siguientes campos:

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `userFeedback` | ENUM | 'LIKE', 'DISLIKE', 'NONE' (Default) |
| `userFeedbackText` | TEXT | Comentario opcional si el usuario marcó DISLIKE |
| `sentimentLabel` | ENUM | 'POSITIVE', 'NEUTRAL', 'NEGATIVE', 'CONFUSED', 'FRUSTRATED' |
| `sentimentScore` | FLOAT | Puntuación de la IA entre -1.0 y 1.0 |
| `usedSourceIds` | JSONB / Array | Lista de UUIDs de los `KnowledgeSource` usados para esa respuesta |



### 4.2 Flujo de Arquitectura
1. **Chat Request**: Usuario envía mensaje → NestJS busca en Pinecone → Gemini genera respuesta RAG → Se guarda mensaje en BD con `usedSourceIds`.
2. **Async Event**: NestJS dispara evento `message.created`.
3. **Sentiment Worker**: Un *Listener* atrapa el evento, envía el texto del usuario a Genkit (`analyzeSentimentFlow`), extrae el sentimiento y actualiza la entidad `Message` en la BD.
4. **Feedback Request**: Usuario pulsa 👎 → Frontend llama a `PATCH /api/v1/interaction/messages/:id/feedback` → NestJS actualiza `userFeedback` en la BD.

## 5. Restricciones Técnicas

* **Librerías Visuales**: El dashboard debe construirse utilizando los componentes de `shadcn/ui` y `Recharts` (para mantener coherencia con la V1).
* **i18n**: Los textos del formulario de feedback ("¿En qué podemos mejorar?") deben estar internacionalizados usando `next-intl` (`messages/es.json`, `messages/en.json`).
* **Genkit Schemas**: La respuesta de Genkit para el análisis de sentimiento debe obligarse a devolver JSON Estructurado (`outputSchema`) para garantizar que la etiqueta y el score puedan mapearse directamente a la base de datos sin parseos manuales complejos.

## 6. Plan de Implementación Propuesto

La funcionalidad se desarrollará y entregará en 3 iteraciones (fases) para aportar valor inmediato:

### Fase 1: Feedback Explícito (Iteración Frontend/Backend Básica)
* Modificar migración de base de datos para agregar las nuevas columnas a la tabla `messages`.
* Crear DTOs y endpoint `PATCH /messages/:id/feedback` en NestJS.
* Implementar componentes UI en el chat de Next.js (Botones 👍/👎 y Modal de reporte).
* *Entregable*: Los usuarios ya pueden calificar respuestas y la data se almacena.

### Fase 2: Trazabilidad y Dashboard (Iteración Data/Admin)
* Modificar el Use Case de `query-assistant` para mapear los UUIDs de los fragmentos recuperados hacia `usedSourceIds`.
* Crear el endpoint `GET /api/v1/stats/knowledge-health` con las agregaciones SQL.
* Construir la pestaña "Salud del Conocimiento" en el Dashboard de Next.js con tablas y gráficos.
* *Entregable*: RRHH puede ver qué documentos fallan basándose en los dislikes.

### Fase 3: IA de Sentimiento (Iteración Async/Genkit)
* Crear el nuevo flujo de Genkit (`analyzeSentimentFlow`) en el backend.
* Configurar `@nestjs/event-emitter` para escuchar nuevos mensajes del usuario.
* Implementar el listener asíncrono que procesa la IA y actualiza el `sentimentLabel`.
* Conectar los datos de sentimiento al Dashboard de RRHH.
* *Entregable*: Flujo completo operando en piloto automático para métricas implícitas.
